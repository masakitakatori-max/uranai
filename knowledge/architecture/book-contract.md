# 書籍準拠アーキテクチャ契約

このアプリの占術精度は、UI や AI の賢さではなく、**書籍の内容をどれだけ実行可能な仕様に変換できているか**で決まる。

ここでは、3 モードを共通の設計原則で扱う。

## 1. 原典を第一級のデータにする

書籍本文、図、表、例題、注釈をすべて「出典つきの知識」として扱う。

- 画像番号
- 章名
- ページ位置
- 例題か表か本文か
- どの判定に使うか

原典は説明文ではなく、後続のルールが参照するデータとする。

## 2. 知識を4層に分ける

### source
書籍の画像、OCR、要約、表、例題。

### rules
「条件 -> 判定 -> 優先順位 -> 出典」の形にした型付きルール。

### resolver
入力から盤を決める決定論ロジック。
ここでは silent fallback を禁止する。

### trace / explainer
なぜその結論になったかを説明する層。
ここは盤の結果を文章化するだけで、判定はしない。

## 3. 断定と近似を分ける

書籍に書いてあることは断定する。
書籍にないが実装都合で補うものは近似として明示する。

- `lookup` は原典由来の断定
- `derived` は補完
- `unresolved` は保留

この3つを混ぜない。

## 4. 3モード共通のアーキテクチャ

### 六壬神課

- `三伝 lookup` を本則の中心にする
- 欠損ケースだけ `derived` で補完する
- `三伝未確定` のときは、第1課などで擬似的に断定しない
- 例題は fixture として固定する

### 断易

- `占的 -> 用神候補 -> 世応 -> 納甲 -> 月破/空亡/日辰関係` の順に判定する
- 相談文の対象を先に推定し、候補の優先順位を決める
- `世応`、`用神`、`原神`、`忌神`、`仇神` を role として区別する
- 宮、爻、関係、旺相休囚死を分離して持つ

### 金口訣

- `地分 -> 月将 -> 貴神/将神 -> 用爻 -> 旺相休囚死` の順に固定する
- 書籍の代表図を fixture で守る
- 近似変換がある箇所は `approximation` として記録する

## 5. ファイル分割方針

### `src/lib/data/`
原典由来の表、fixture、出典 index。

### `src/lib/rules/`
生剋、関係、優先順位、選定ルール。

### `src/lib/resolvers/`
入力から盤を作る純関数。

### `src/lib/explainers/`
盤から自然文を作る層。

### `src/lib/traces/`
どの規則が効いたかの説明用データ。

### `src/lib/*.test.ts`
書籍 fixture と invariant を固定する回帰テスト。

## 6. 本準拠の判定基準

次の条件を満たすほど、本に準拠していると言える。

1. 書籍例が fixture で再現できる
2. 例外規則が例外として分離されている
3. 補完が補完として明示されている
4. silent fallback がない
5. すべての結論に source ID がある
6. 盤の trace を見れば、どのルールで決まったか追える

## 7. 現状の評価

- 六壬神課: lookup を中心に、欠損だけ補完する形になっている
- 断易: 明示ルールはまだ薄く、近似が残る
- 金口訣: 3 モードの中では最も書籍寄りだが、まだ representative rules が中心

## 8. 今後の優先順位

1. 断易の用神選定を候補ベースに細分化する
2. 金口訣の近似変換を approximation として隔離する
3. 六壬神課の derived 部分を lookup と同格に見せない
4. すべてのモードで `未確定` の扱いを統一する
